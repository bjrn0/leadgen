# Architecture — LeadGen

> Generated by bmad-document-project (deep scan) on 2026-05-29. Single-part `web` project.

This document describes the **architecture as implemented in code today**, then contrasts it with the **planned/target architecture** captured in `specs/`. Keeping these separate is essential: most of the system in the specs is not yet built.

---

## 1. Implemented architecture (current reality)

A client-rendered Next.js dashboard that is, in effect, a high-fidelity interactive prototype of the product. No server-side data, no persistence, no external calls beyond optional Clerk auth.

### Shape

```text
Browser
 └─ Next.js App Router (app/src/app)
     ├─ layout.tsx      → optional ClerkProvider wrapper + Toaster
     └─ page.tsx        → "use client" dashboard shell
         ├─ SidebarProvider / AppSidebar  → view switch (useState)
         ├─ LeadGenerationView  → renders hardcoded `leads[]`
         └─ MonitoringView      → renders hardcoded `accounts[]`
```

### Key decisions & patterns

- **App Router, but a single route.** Only `app/src/app/page.tsx` (`/`) exists. The two "views" (Lead Generation, Monitoring) are switched with a `useState<ViewType>` in `page.tsx` — there is **no** `/lead-generation` or `/monitoring` route. (The `specs/ui-mapping.md` route map and README's "route-based pages" are aspirational.)
- **All client components.** Every component is `"use client"`. There are no server components, server actions, route handlers, or API routes (`app/api/` does not exist).
- **Mock data is co-located in views.** `leads[]` lives in [lead-generation-view.tsx](../app/src/app/views/lead-generation-view.tsx); `accounts[]` lives in [monitoring-view.tsx](../app/src/app/views/monitoring-view.tsx). Filtering/pagination operate purely in-memory (`useMemo`, `.slice(0, 5)`).
- **Actions are simulated.** Every action button (Add to Watchlist, Push to CRM, Suppress, Start Outreach, Create Task, Add Source, Create ICP Rule, Apply Filters) calls `toast.*` from `sonner`. None emit a real backend event.
- **Auth is optional and off by default.** [auth-mode.ts](../app/src/lib/auth-mode.ts) reads `NEXT_PUBLIC_AUTH_MODE` (default `dev-bypass`). In dev-bypass mode `layout.tsx` skips `ClerkProvider` entirely. A static `fallbackUser` is shown in the sidebar footer.
- **Theming via CSS variables.** Brand colors (`--brand`, `--brand-light`, `--brand-border`) are referenced inline and defined in `globals.css`.

### Technology stack

See the table in [project-overview.md](./project-overview.md#tech-stack-implemented).

### State management

Local React only: `useState`, `useMemo`, `useEffect` (e.g. Escape-key handlers for modals). The shadcn `SidebarProvider` supplies sidebar open/collapsed context. No Redux/Zustand/Context-based global store.

### Entry points

- `app/src/app/layout.tsx` — root layout, metadata, conditional Clerk wrapper, global `Toaster`.
- `app/src/app/page.tsx` — `DashboardPage`, the only page.

## 2. Planned / target architecture (from specs — NOT yet implemented)

The README and `specs/` describe a much larger AWS-native engine. None of the following exists in code yet; it is documented here so contributors understand the intended destination. Source of truth: [../specs/architecture.md](../specs/architecture.md) and [../_bmad-output/specs/spec-leadgen/SPEC.md](../_bmad-output/specs/spec-leadgen/SPEC.md).

```text
RDS source registry + ICP rules + watchlists
 -> EventBridge Scheduler -> Go source adapters / managed extraction
 -> EventBridge + SQS -> hot-path detector
 -> Step Functions cold path -> S3 + Nebius AI Studio + Cohere + Pinecone
 -> RDS signals/timelines/contacts/alerts
 -> dashboard + notifications + CRM/n8n actions
```

- **Intended repo layout (README):** `app/` (this UI), `database/` (SQL schema + bootstrap), `services/ingest/` (Go polling/normalization), `services/worker/` (Go deep processing), `docs/`.
- **Intended data plane:** RDS Postgres + pgvector as system of record; S3 for raw artifacts/audio/transcripts; Pinecone for semantic search; Cohere rerank; Nebius AI Studio for structured LLM extraction.
- **Intended orchestration:** EventBridge Scheduler, EventBridge, SQS, Lambda, Step Functions (hot path / cold path split).
- **Intended ingestion:** structured feeds/APIs (RSS, YouTube, Greenhouse, Lever), managed extraction (Firecrawl, Bright Data, WebShare), multimedia transcription.

### Notable divergences to resolve

- **Notifications:** `specs/` say Resend (email) + webhook; `.env.example` declares Slack + Telegram. The implemented UI shows generic "email + webhook" routing icons. Channel choice is unsettled.
- **`db:init` script:** root `package.json` defines `"db:init": "tsx database/scripts/init.ts"`, but `database/` does not exist — the script will fail until the schema package is created.
- **README setup commands** reference `npm run dev:web`; the actual script is `npm run dev`. See [development-guide.md](./development-guide.md).

## 3. Integration points

**Current:** none (no part-to-part integration; single frontend). The only external dependency that can activate at runtime is Clerk, and only when `NEXT_PUBLIC_AUTH_MODE != dev-bypass`.

**Planned:** the UI would call the Go services / an API layer that reads RDS and returns signals, evidence, contacts, and summaries. That contract does not exist yet.
