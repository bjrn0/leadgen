# Project Overview — LeadGen

> Generated by bmad-document-project (deep scan) on 2026-05-29. Primary AI-context entry point: [index.md](./index.md).

## Purpose

LeadGen is an internal **go-to-market (GTM) intelligence engine** for sales, business-development, and account-research teams. It is designed to turn public internet signals into evidence-backed sales actions across two pillars:

- **Lead Generation** — discover new companies/people matching an ICP.
- **Account Monitoring** — continuously track watchlisted accounts and summarize what changed.

The canonical product contract lives in [../_bmad-output/specs/spec-leadgen/SPEC.md](../_bmad-output/specs/spec-leadgen/SPEC.md); long-form product/architecture notes are in [../specs/](../specs/).

## ⚠️ Implementation status (read this first)

**What exists in code today is a frontend prototype only.** The repository currently contains a Next.js dashboard (`app/`) that renders the two-pillar UI against **hardcoded mock data**. There is no backend, no database access, no API layer, and no ingestion/AI pipeline implemented yet.

The broader system described in the README and `specs/` (AWS-native engine, Go ingestion/worker services, RDS Postgres, S3, Pinecone, etc.) is the **target architecture**, not the current state. See [architecture.md](./architecture.md) for the explicit "implemented vs planned" split.

| Aspect | Documented intent (specs/README) | Actually in the repo |
| --- | --- | --- |
| Repository shape | Monorepo: `app`, `database`, `services/ingest` (Go), `services/worker` (Go), `docs` | Only `app/` (Next.js) and `specs/` exist |
| Backend / ingestion | Go services + AWS EventBridge/SQS/Lambda/Step Functions | None |
| Database | RDS Postgres + pgvector | None (env var declared, `db:init` script points at a missing `database/` folder) |
| Data in UI | Live signals/evidence from RDS | Hardcoded mock arrays inside view components |
| Actions (CRM, outreach, suppress) | Real backend events | `sonner` toast notifications only |
| Notifications | Resend email + webhook (specs) / Slack + Telegram (`.env.example`) | Not wired to any code |
| Auth | Clerk | Clerk wired, but defaults to `dev-bypass` (no auth) |
